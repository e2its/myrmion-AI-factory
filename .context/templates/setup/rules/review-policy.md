# Review Policy Configuration

> **Status:** Active  
> **Enforcement:** Mandatory for all features before QA verification  
> **Agent:** IMPLEMENT.AGENT.MD (🔍 REVIEW hat)

---

## 🎯 Review Philosophy

Peer review acts as a **quality gate** between implementation (💻 DEV hat) and certification (🧪 QA). The goal is to catch architectural violations, security issues, and code quality problems early, before they consume QA resources.

### Core Principles

1. **AI-Driven:** Reviews are performed automatically by AI (no human assignment required)
2. **Sequential Blocking:** Features cannot proceed to QA until review is approved
3. **Configurable Strictness:** Review depth adapts to environment and risk level
4. **Retry Escalation:** After 3 rejections, automatic escalation to Architect for conflict resolution
5. **Override Capability:** Emergency bypass available with mandatory justification

---

## 📊 Review Levels

### STRICT (Production Code)
**When:** Features targeting `main` branch or `production` environment  
**Scope:** Deep analysis of all code quality metrics  
**Enforcement:**
- ❌ **BLOCKER:** Architecture violations, security issues, missing tests, hardcoded config
- ⚠️ **WARNING:** Code smells, missing error handling, poor naming, missing traceability
- 💡 **NITPICK:** Style suggestions, readability improvements

**Exclusions:** None (all files reviewed)

### STANDARD (Staging/Development)
**When:** Features targeting `staging` or `development` environments  
**Scope:** Moderate analysis focusing on critical issues  
**Enforcement:**
- ❌ **BLOCKER:** Architecture violations, security issues, hardcoded config
- ⚠️ **WARNING:** Missing tests, poor error handling
- 💡 **NITPICK:** Disabled (not shown in report)

**Exclusions:** Test fixtures, mock data

### RELAXED (Prototypes/Experiments)
**When:** Features tagged as `prototype` or `experimental`  
**Scope:** Minimal analysis, only critical violations  
**Enforcement:**
- ❌ **BLOCKER:** Security issues, hardcoded secrets
- ⚠️ **WARNING:** Disabled (not shown in report)
- 💡 **NITPICK:** Disabled (not shown in report)

**Exclusions:** Test fixtures, mock data, generated code

## 🚫 Review Exclusions

Files matching these patterns are **automatically skipped** during peer review:

```json
{
  "review_exclusions": [
    "**/*.mock.ts",
    "**/*.mock.js",
    "**/*.mock.py",
    "**/fixtures/**",
    "**/test-data/**",
    "**/__snapshots__/**",
    "**/*.generated.ts",
    "**/*.generated.js",
    "**/*.generated.py",
    "**/dist/**",
    "**/build/**",
    "**/.next/**",
    "**/coverage/**",
    "**/*.min.js",
    "**/*.bundle.js"
  ]
}
```

**Rationale:**
- **Mock files:** Test utilities, not production code
- **Fixtures:** Static test data
- **Generated code:** Auto-generated by tools (GraphQL codegen, Prisma, etc.)
- **Build artifacts:** Compiled output, not source
- **Minified files:** Already optimized, not reviewable

## 🔄 Retry Policy & Escalation

### Max Retry Attempts: 3

After **3 rejections**, the feature automatically escalates to the Architect:

```
Attempt 1: DEV → REVIEW (REJECT) → DEV fix → REVIEW
Attempt 2: DEV → REVIEW (REJECT) → DEV fix → REVIEW  
Attempt 3: DEV → REVIEW (REJECT) → DEV fix → REVIEW
Attempt 4: ⛔ MAX RETRIES → AUTO-ESCALATE to `/BLUEPRINT --review-conflict`
```

### Escalation Triggers

Automatic escalation occurs when:
- `review_attempt_count >= 3` in `dev_plan.md` frontmatter
- Same BLOCKER appears in 3 consecutive reviews (systematic issue)
- Architect receives notification with full review history

### Architect Conflict Resolution

When escalated, ARCH agent must:
1. Read all 3 review reports
2. Analyze pattern of rejections
3. Determine root cause:
   - Design is unimplementable → Revise design
   - Requirements are ambiguous → Escalate to PO
   - Review is too strict → Override with justification
4. Make **binding decision** (approve/reject/redesign)
5. Reset retry counter if design changes

## 🚨 Override Mechanism

### When to Use Override

- **Emergency hotfix:** Production outage requires immediate deploy
- **False positive:** Review incorrectly flagged valid code
- **Architect decision:** After conflict resolution, ARCH decides to bypass

### Override Requirements

1. **Mandatory Justification:** Minimum 50 characters explaining why
2. **ARCH Notification:** Automatic alert sent to Architect
3. **Audit Trail:** Override logged in `OVERRIDE_LOG.md` and workflow log
4. **QA Flag:** QA verification sees override flag in report
5. **SEC Review:** Security audit includes override in compliance check

### Override Command

```
/IMPLEMENT --override {{FEATURE_ID}} "Justification: [detailed reason]"
```

**Example:**
```
/IMPLEMENT --override USR-001 "Production payment gateway down. Code has minor style issues but is functionally correct and security-vetted. Deploying under incident response protocol IR-2026-01."
```

**Output:**
- Updates review report: `status: OVERRIDDEN`
- Updates dev plan: `review_status: APPROVED (OVERRIDDEN)`
- Creates override log entry
- Unlocks QA verification

## 📋 Review Checklist (Automatic)

The PEER_REVIEW agent automatically validates:

### 1. Architecture Compliance ✅
- [ ] Domain layer does not import Infrastructure
- [ ] Application layer uses interfaces (not concrete implementations)
- [ ] Infrastructure properly separated from business logic
- [ ] Dependency flow follows design.md layer rules

### 2. Protected Code Policy ✅
- [ ] No RED ZONE modifications without ADR
- [ ] No changes to `node_modules`, `venv`, framework code
- [ ] Legacy code modifications documented in design.md Section 4 (ADR)

### 3. Configuration Security ✅
- [ ] No hardcoded URLs, API keys, credentials
- [ ] All config externalized to `.env` or `/config/system_resources.json`
- [ ] No IP addresses hardcoded in source

### 4. Traceability ✅
- [ ] All feature files have `// Ref: {{FEATURE_ID}}` header
- [ ] Files match design.md Section 3 (Inventory)
- [ ] No unplanned artifacts created

### 5. Test Coverage ✅
- [ ] Unit tests exist for new business logic
- [ ] Test files follow naming convention (`*.test.ts`, `*.spec.ts`)
- [ ] AAA pattern (Arrange-Act-Assert) followed

### 6. Path References Policy ✅ (CRITICAL)
- [ ] **BLOCKER:** No absolute paths in ANY source file (e.g., `/home/`, `/Users/`, `C:\`, `/opt/`)
- [ ] All file references use relative paths or module aliases (`@/`, `../`)
- [ ] System path exceptions (e.g., `/tmp/`) are documented with inline comments
- [ ] Applies to: source code, tests, config files, scripts

### 7. Error Handling ✅
- [ ] No empty catch blocks
- [ ] No generic error swallowing (`catch (e) {}`)
- [ ] Structured logging (no `console.log` in production paths)

### 7. Code Quality (Strictness-Dependent) ⚠️
- [ ] Naming conventions followed (camelCase/snake_case per language)
- [ ] No code duplication (DRY principle)
- [ ] Functions are single-responsibility (SOLID)
- [ ] Comments explain "why", not "what"

## 🔗 Integration Points

### DEV → IMPLEMENT (Unified Build)
When DEV plan is ready, execute `/IMPLEMENT --build {{FEATURE_ID}}`:
- 💻 DEV implements (TDD) → 🔍 REVIEW verifies (governance + quality) → 🛡️ SEC scans (SAST) per phase
- Review is inline within `/IMPLEMENT --build` (🔍 REVIEW hat)
- Seamless multi-hat workflow

### REVIEW → QA (Sequential Blocking)
When QA executes `/QA --verify {{FEATURE_ID}}`:
- Checks for `review_report.md` with `status: APPROVED`
- Blocks if review not completed
- Allows override if `override_justification` exists

### REVIEW → ARCH (Escalation)
When retry limit exceeded:
- Automatically calls `/BLUEPRINT --review-conflict {{FEATURE_ID}}`
- BLUEPRINT receives full review history
- BLUEPRINT makes binding decision

## 📈 Metrics & Analytics

### Review Performance Tracking

Track in `docs/project_log/workflow_log.json`:

| Metric | Description | Target |
|--------|-------------|--------|
| **Pass Rate** | % of features approved on 1st attempt | >80% |
| **Avg Attempts** | Average retries before approval | <1.5 |
| **Escalation Rate** | % of features escalated to ARCH | <5% |
| **Override Rate** | % of features overridden | <2% |
| **Common Blockers** | Top 3 rejection reasons | Track patterns |

### Continuous Improvement

Monthly review of metrics to:
- Identify systematic issues (e.g., if 80% fail on "missing tests" → improve DEV training)
- Adjust review strictness if too many false positives
- Update exclusion patterns if irrelevant files flagged
